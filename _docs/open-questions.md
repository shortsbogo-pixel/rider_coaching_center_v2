# 구현 전 확인 질문

## 데이터/엑셀

1. 파일명 `YYYY_MM-W`에서 화면 주차 라벨은 `2026년 5월 4주차`, `5월4주차`, `2026-W22` 중 무엇을 표준으로 쓸까요?
2. 주차 기간은 기획 시안처럼 수요일 시작~다음 주 화요일 종료로 확정인가요?
3. 같은 주차를 재업로드할 때 기본 동작은 기존 데이터 교체, 새 버전 저장, 업로드 차단 중 무엇인가요?
4. `오더별 상세 내역서` 외 다른 시트는 MVP에서 완전히 제외해도 될까요?
5. 샘플에 없는 `완료건수`는 오더 1행 = 1건으로 확정해도 될까요?
6. `배달타입` 값 `0`, `멀티배달5`, 빈 값은 모두 `확인필요`로 분리하면 될까요?
7. `피크타임` 빈 값이 있는 행은 저장하되 검수 이슈로 표시할까요, 아니면 저장 차단할까요?
8. 주소/지역 텍스트는 원문 저장해도 되는지, 마스킹 또는 별도 보안 기준이 필요한가요?

## 라이더 식별/계정

1. 라이더 계정의 기본 매칭 키는 이름, 전화번호, 별도 rider_id 중 무엇인가요?
2. 엑셀의 `이름` 값에 붙은 숫자 suffix를 그대로 표시해도 되나요, 아니면 화면에서는 이름을 마스킹/정리해야 하나요?
3. 동명이인 또는 같은 이름의 숫자 suffix 차이는 어떻게 처리할까요?
4. 회원가입 후 관리자 승인 방식은 필수인가요, 아니면 관리자가 계정을 선생성하나요?
5. 라이더 비밀번호 초기화/변경은 MVP에 포함하나요?
6. inactive 계정은 로그인 자체 차단, 또는 로그인 후 비활성 안내 중 무엇이 맞나요?

## 코칭 메시지

1. 자동 메시지는 규칙 기반으로 먼저 만들고, AI 생성은 이후 확장으로 미뤄도 될까요?
2. 관리자 수정 메시지는 저장 즉시 라이더 노출인가요, 아니면 `visible_to_rider` ON을 별도로 눌러야 하나요?
3. 노출 OFF 상태일 때 라이더에게 자동 메시지를 보여줄까요, 아니면 “아직 코칭 메시지가 없습니다”로 표시할까요?
4. 관리자 내부 메모는 주차별/라이더별로 저장하면 충분한가요?
5. 템플릿 관리는 MVP 필수인가요, 아니면 자동 메시지 규칙에 하드코딩 후 추후 관리 화면을 만들까요?
6. 코칭 메시지 발송 기능은 복사 버튼까지만 MVP에 포함하고 실제 문자/카톡 발송은 제외해도 될까요?

## 점수/지표

1. 배차 친화 점수의 공식은 Ver.1 기준을 그대로 쓸까요, Ver.2용으로 단순화할까요?
2. 원시 엑셀에 거절/무시율이 없는데, 해당 항목은 점수에서 제외해도 될까요?
3. 등급 기준은 Ver.1의 완료건수 기준(S/A/B/C/관리대상)을 유지하나요?
4. “출근일”은 완료 오더가 1건 이상 있는 날짜 수로 계산하면 될까요?
5. 예상 정산은 `정산금액` 합계로 표시해도 되는지, 아니면 MVP에서는 숨길까요?

## 관리자 화면

1. 관리자 대시보드 첫 화면의 최우선 KPI는 완료건수, 운영 라이더 수, 검수 이슈, 코칭 필요 수 중 무엇인가요?
2. 검수 화면에서 관리자가 직접 값 보정까지 해야 하나요, 아니면 초기에는 이슈 확인만 하나요?
3. 라이더명 매칭 보정 UI는 MVP에 포함해야 하나요?
4. `더보기`에 반드시 들어갈 항목은 계정 관리, 문의번호/공지 외에 무엇인가요?

## 라이더 화면/지도

1. `내지도`는 MVP에서 실제 지도 연동이 필수인가요, 아니면 주소 기반 목록/간이 지도 placeholder로 시작해도 될까요?
2. Google Maps 등 외부 지도 API 사용 예산/키 관리 정책이 있나요?
3. 라이더 `내오더`에서 정산금액을 보여줘도 되나요?
4. 라이더에게 검수 상태나 데이터 누락 경고를 어느 수준까지 보여줄까요?
