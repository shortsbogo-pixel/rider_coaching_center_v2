# 배차 포지션 추천 기능 설계 문서

## 전제

이 문서는 라이더 코칭센터 Ver.2에 향후 “배차 포지션 추천” 기능을 추가하기 위한 설계 초안이다.

이번 단계에서는 기능 구현을 하지 않는다. Google Maps API, DB/Supabase, 외부 API, GPS, 현재 위치, 실시간 콜 데이터, 실시간 관제 기능을 연결하지 않는다.

현재 앱은 쿠팡플러스 전용이다. 기준 데이터는 관리자가 업로드하고 반영한 주차별 확정 엑셀 데이터다.

## 1. 기능 목적

배차 포지션 추천 기능의 목적은 라이더가 선택 주차, 요일, 시간대 기준으로 배차 가능성이 높았던 대기 권역을 참고할 수 있게 하는 것이다.

이 기능은 실시간 배차 보장 기능이 아니다. 과거 업로드 데이터 기반의 코칭 기능이다.

라이더에게 주는 핵심 도움:

- 특정 요일/시간대에 픽업이 많았던 권역을 참고한다.
- 배달 완료 후 어느 권역으로 복귀할지 판단하는 데 도움을 받는다.
- 본인 과거 수행 경험과 전체 픽업 밀집 권역을 비교해 다음 대기 위치를 정한다.

명확히 제외하는 것:

- 현재 위치 추적
- 실시간 콜 진행 상태 확인
- 실시간 배차 예측
- 관제 화면
- 고객 위치 표시
- 배차 보장 문구

## 2. 사용자별 기능

### 라이더

라이더 화면에서는 “오늘 어디서 대기할까”를 짧고 쉽게 보여준다.

라이더 기능 후보:

- 오늘 어디서 대기할까 카드
- 시간대별 추천 대기권역
- 요일별 추천 대기권역
- 배달 완료 후 복귀 추천 권역
- 본인 과거 수행 경험 기준 권역 표시

라이더에게 보여줄 기준:

- 선택 주차 업로드 데이터 기준
- 본인 `rider_id` 기준 수행 경험
- 같은 요일/시간대 픽업이 많았던 권역
- 배달 완료 후 가까운 픽업 밀집 권역

라이더에게 보여주지 않을 것:

- 타 라이더 상세 오더
- 고객 상세 위치
- 실시간 현재 위치
- 실시간 배차 가능 여부

### 관리자

관리자 화면에서는 주차별/시간대별/라이더별 권역 분석을 코칭 참고 자료로 본다.

관리자 기능 후보:

- 전체 픽업 밀집 권역 분석
- 라이더별 활동 권역 분석
- 시간대/요일별 픽업 TOP 지역
- 피크타임별 추천 대기권역
- 코칭 메시지 작성 참고
- 관리자 대시보드에 최신 업로드 주차 기준 픽업 밀집 분석 카드 추가(29단계 후보)

관리자 활용 예:

- 특정 라이더가 강한 권역과 약한 권역을 확인한다.
- 특정 시간대에 픽업이 반복되는 지역을 찾는다.
- 코칭 메시지에 “다음 주에는 Post_Dinner에 ○○권역 대기 시도”처럼 참고 문구를 작성한다.

## 3. 데이터 기준

배차 포지션 추천의 1차 데이터는 정규화된 업로드 데이터다.

필요 필드:

| 필드 | 설명 |
| --- | --- |
| `week_code` | 주차 코드. 예: `2026_05-4` |
| `rider_id` | 라이더 고정 ID |
| `rider_name` | 업로드 시점의 라이더명 |
| `pickup_name` | 픽업 상점명 또는 픽업지명 |
| `pickup_address` | 픽업 상세 주소 |
| `pickup_area` | 픽업 권역 또는 행정 구역 |
| `dropoff_region` | 배달 지역 상위 구분 |
| `dropoff_dong` | 배달 동 단위 정보 |
| `completed_at` | 배달 완료 시각 |
| `assigned_at` | 배정 시각 |
| `peak_time` | 피크타임 구간. 예: `Post_Lunch`, `Post_Dinner` |
| `delivery_type` | 단건/멀티배달 계열 |
| `order_no` | 주문번호 |

파생 가능한 값:

- 요일
- 시간대
- 피크타임
- 픽업지역별 완료 콜 수
- 배달동별 완료 콜 수
- 라이더별 수행 권역
- 시간대/요일별 반복 발생 권역

## 4. 픽업 / 배달 지도 표시 정책

개인정보 보호와 오해 방지를 위해 픽업지와 배달지는 표시 정책을 다르게 둔다.

| 구분 | 표시 기준 | 표시 방식 |
| --- | --- | --- |
| 픽업지 | 상세 주소 기준 표시 가능 | 상점/픽업지 마커 또는 권역 묶음 |
| 배달지 | 동 단위 중심 표시 | 동 중심점, 영역, 흐릿한 권역 표시 |
| 경로선 | 실제 주행 경로 아님 | 픽업 권역과 배달 동을 잇는 참고선 |

주의:

- 배달지를 고객 실제 위치처럼 표시하지 않는다.
- 배달 상세 주소를 지도에 찍지 않는다.
- 경로선은 실제 주행 경로가 아니라 참고선으로 표시한다.
- 지도는 코칭 참고용이며 실시간 위치 지도처럼 보이면 안 된다.

## 5. 추천 알고리즘 초안

### 1차 알고리즘

DB나 외부 지도 API 없이 업로드 데이터만으로 가능한 수준이다.

기준:

- 선택 주차
- 요일
- 시간대 또는 `peak_time`
- 픽업지역 기준 완료 콜 수

방식:

1. 선택 주차의 오더를 필터링한다.
2. 선택 요일과 시간대에 해당하는 오더를 필터링한다.
3. 픽업지역별 완료 콜 수를 집계한다.
4. 완료 콜 수 기준 TOP 5를 정렬한다.
5. 동일 시간대에 반복 발생한 지역을 우선 표시한다.

출력:

- 추천 1순위 권역
- 보조 추천 권역 2~3개
- 지난주 같은 시간대 완료 콜 수
- 추천 이유

### 2차 알고리즘

라이더 개인 경험과 복귀 판단을 반영한다.

추가 기준:

- 라이더 본인 과거 수행 경험
- 배달 완료 후 가까운 픽업 밀집 권역
- 피크타임 가중치
- 멀티배달 발생 권역 가중치

방식:

1. 전체 픽업 밀집 TOP 지역을 계산한다.
2. 본인 `rider_id`의 수행 권역을 별도로 계산한다.
3. 본인 수행 경험이 있는 권역에 기본 신뢰도를 부여한다.
4. 피크타임과 멀티배달 발생 빈도에 가중치를 준다.
5. 배달 완료 권역에서 가까운 픽업 밀집 권역을 후보로 표시한다.

주의:

- 실제 거리 계산은 Google Maps 연동 전까지 정확한 거리값으로 표현하지 않는다.
- “가까운” 표현은 좌표/거리 계산이 붙기 전에는 “복귀 후보” 또는 “인접 권역 후보” 수준으로 제한한다.

### 3차 알고리즘

DB와 지도 연동 이후의 고도화 단계다.

추가 후보:

- DB 도입 후 누적 주차 비교
- Google Maps 거리 계산
- 권역별 점수화
- 추천 신뢰도 표시
- 주차별 추세 비교

점수 예시:

```text
추천 점수 =
  픽업 완료 콜 수 점수
+ 같은 요일/시간대 반복 점수
+ 피크타임 가중치
+ 멀티배달 발생 가중치
+ 본인 수행 경험 가중치
- 거리/복귀 부담 점수
```

신뢰도 예시:

- 높음: 같은 주차/요일/시간대에 반복 픽업이 많고 본인 경험도 있음
- 보통: 전체 픽업은 많지만 본인 경험은 적음
- 참고: 데이터 수가 적어 참고용으로만 표시

## 6. UI 문구 기준

### 사용할 문구

- 선택 주차 업로드 데이터 기준
- 지난주 같은 시간대 픽업이 많았던 권역입니다
- 배차 가능성이 높았던 대기 권역 참고
- 실시간 위치가 아닙니다
- 배달지는 개인정보 보호를 위해 동 단위로 표시됩니다
- 과거 운행 데이터 기준 코칭 참고 정보입니다
- 이 권역은 배차를 보장하지 않습니다

### 사용하지 말 문구

- 실시간 배차 추천
- 현재 위치
- 관제
- 라이브 콜
- 고객 위치
- 여기 있으면 무조건 배차
- 지금 여기로 이동하세요
- 배차 확정
- 실시간 대기 포인트

문구 원칙:

- 배차 보장처럼 들리는 표현을 쓰지 않는다.
- 현재 위치를 수집하는 것처럼 보이는 표현을 쓰지 않는다.
- 고객 위치를 표시하는 것처럼 보이는 표현을 쓰지 않는다.
- 라이더가 참고해서 판단하는 코칭 정보임을 명확히 한다.

## 7. 라이더 화면 UI 초안

카드명:

- 어디서 대기할까?

표시 항목:

| 항목 | 예시 |
| --- | --- |
| 추천 1순위 권역 | 용전동 픽업권역 |
| 추천 이유 | 지난주 같은 시간대 픽업이 가장 많았던 권역입니다. |
| 시간대 | Post_Dinner |
| 요일 | 금요일 |
| 지난주 기준 완료 콜 수 | 18건 |
| 보조 추천 권역 | 홍도동, 가양동, 성남동 |
| 안내 문구 | 선택 주차 업로드 데이터 기준입니다. 실시간 위치가 아닙니다. |

라이더 카드 구성 예:

```text
어디서 대기할까?

추천 1순위
용전동 픽업권역

지난주 금요일 Post_Dinner에 픽업이 많았던 권역입니다.

보조 추천
홍도동 · 가양동 · 성남동

안내
선택 주차 업로드 데이터 기준입니다.
배차를 보장하지 않으며 실시간 위치가 아닙니다.
```

라이더 화면 배치 후보:

- 내지도 화면 상단 카드
- 내현황의 오늘의 페이스 체크 아래 보조 카드
- 내코칭의 실천 미션 하단 참고 카드

1차 적용 우선 후보:

- 내지도 화면에 “추천권역 카드” 추가
- 지도 실제 연동 전에는 리스트/카드 형태로 먼저 제공

## 8. 관리자 화면 UI 초안

카드명:

- 시간대별 픽업 밀집 분석
- 라이더별 배차 포지션 분석

표시 항목:

| 항목 | 설명 |
| --- | --- |
| 주차 선택 | 분석할 업로드 주차 |
| 요일 선택 | 월~일 또는 전체 |
| 시간대 선택 | Breakfast, Lunch_Peak, Post_Lunch, Dinner_Peak, Post_Dinner |
| 라이더 선택 | 전체 또는 특정 라이더 |
| 픽업지역 TOP 5 | 픽업 완료 콜 수 기준 |
| 배달동 TOP 5 | 동 단위 집계 |
| 추천 대기권역 | 선택 조건 기준 추천 |
| 코칭 메모 | 관리자 코칭 메시지 작성 참고 |

관리자 카드 구성 예:

```text
시간대별 픽업 밀집 분석

주차: 2026_05-4
요일: 금요일
시간대: Post_Dinner

픽업지역 TOP 5
1. 용전동 18건
2. 가양동 14건
3. 홍도동 11건

추천 대기권역
용전동 → 가양동 권역 참고

코칭 메모
금요일 저녁에는 용전동 권역 대기 전략을 안내할 수 있습니다.
```

관리자 화면 주의:

- 라이더 선택은 관리자 화면에만 둔다.
- 라이더 화면에는 타 라이더 선택 UI를 만들지 않는다.
- 배달동은 동 단위로만 표시한다.

## 9. 지도 연동 단계

| 단계 | 내용 | 구현 범위 |
| --- | --- | --- |
| 1단계 | 문서화 | 현재 단계. 기능 구현 없음 |
| 2단계 | 데이터 타입 추가 | 픽업/배달 권역 분석용 타입과 순수 집계 함수 설계 |
| 3단계 | 내지도 추천권역 카드 | 지도 API 없이 카드/리스트로 추천권역 표시 |
| 4단계 | 관리자 픽업 밀집 분석 | 관리자 화면에 TOP 지역/요일/시간대 분석 추가 |
| 5단계 | 좌표 캐시 설계 | 픽업 주소 좌표 변환 결과 캐시 구조 설계 |
| 6단계 | Google Maps 실제 연동 | 공식 API 검토 후 지도 표시/거리 계산 도입 |

Google Maps 실제 연동 전 확인:

- API 비용
- 주소 지오코딩 제한
- 키 보안과 환경변수 관리
- 좌표 캐시 저장 위치
- 배달지 동 단위 표시 정책
- 실시간 위치처럼 보이지 않는 UI 문구

## 10. 개인정보 / 운영 주의사항

- 배달지는 동 단위만 사용한다.
- 고객 주소 추정을 금지한다.
- 원천 엑셀 파일 저장을 금지한다.
- 실시간 위치 추적을 금지한다.
- 라이더는 본인 데이터만 조회한다.
- 관리자는 전체 권역 분석 가능하되 원천 개인정보를 노출하지 않는다.
- 지도에 표시되는 정보는 과거 업로드 데이터 기반의 코칭 참고 정보다.
- 배차 가능성이 높았던 권역이라는 표현은 가능하지만 배차 보장 표현은 금지한다.
- 정산 데이터와 직접 연동하지 않는다.
- 현재 위치/GPS 권한 요청을 하지 않는다.

운영 전 결정할 항목:

| 항목 | 결정 필요 |

- 제외 범위: Google Maps API, DB/Supabase, 외부 API, GPS/현재 위치, 실시간 콜 진행 데이터
