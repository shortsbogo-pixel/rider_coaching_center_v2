# 배차 포지션 추천 기능 설계 문서

## 전제

이 문서는 라이더 코칭센터 Ver.2에 향후 “배차 포지션 추천” 기능을 추가하기 위한 설계 초안이다.

이번 단계에서는 기능 구현을 하지 않는다. Google Maps API, DB/Supabase, 외부 API, GPS, 현재 위치, 실시간 콜 데이터, 실시간 관제 기능을 연결하지 않는다.

현재 앱은 쿠팡플러스 전용이다. 기준 데이터는 관리자가 업로드하고 반영한 주차별 확정 엑셀 데이터다.

## 1. 기능 목적

배차 포지션 추천 기능의 목적은 라이더가 선택 주차, 요일, 시간대 기준으로 배차 가능성이 높았던 대기 권역을 참고할 수 있게 하는 것이다.

이 기능은 실시간 배차 보장 기능이 아니다. 과거 업로드 데이터 기반의 코칭 기능이다.

라이더에게 주는 핵심 도움:

- 특정 요일/시간대에 픽업이 많았던 권역을 참고한다.
- 배달 완료 후 어느 권역으로 복귀할지 판단하는 데 도움을 받는다.
- 본인 과거 수행 경험과 전체 픽업 밀집 권역을 비교해 다음 대기 위치를 정한다.

명확히 제외하는 것:

- 현재 위치 추적
- 실시간 콜 진행 상태 확인
- 실시간 배차 예측
- 관제 화면
- 고객 위치 표시
- 배차 보장 문구

## 2. 사용자별 기능

### 라이더

라이더 화면에서는 “오늘 어디서 대기할까”를 짧고 쉽게 보여준다.

라이더 기능 후보:

- 오늘 어디서 대기할까 카드
- 시간대별 추천 대기권역
- 요일별 추천 대기권역
- 배달 완료 후 복귀 추천 권역
- 본인 과거 수행 경험 기준 권역 표시

라이더에게 보여줄 기준:

- 선택 주차 업로드 데이터 기준
- 본인 `rider_id` 기준 수행 경험
- 같은 요일/시간대 픽업이 많았던 권역
- 배달 완료 후 가까운 픽업 밀집 권역

라이더에게 보여주지 않을 것:

- 타 라이더 상세 오더
- 고객 상세 위치
- 실시간 현재 위치
- 실시간 배차 가능 여부

### 관리자

관리자 화면에서는 주차별/시간대별/라이더별 권역 분석을 코칭 참고 자료로 본다.

관리자 기능 후보:

- 전체 픽업 밀집 권역 분석
- 라이더별 활동 권역 분석
- 시간대/요일별 픽업 TOP 지역
- 피크타임별 추천 대기권역
- 코칭 메시지 작성 참고
- 관리자 대시보드에 최신 업로드 주차 기준 픽업 밀집 분석 카드 추가(29단계 후보)

관리자 활용 예:

- 특정 라이더가 강한 권역과 약한 권역을 확인한다.
- 특정 시간대에 픽업이 반복되는 지역을 찾는다.
- 코칭 메시지에 “다음 주에는 Post_Dinner에 ○○권역 대기 시도”처럼 참고 문구를 작성한다.

## 3. 데이터 기준

배차 포지션 추천의 1차 데이터는 정규화된 업로드 데이터다.

필요 필드:

| 필드 | 설명 |
| --- | --- |
| `week_code` | 주차 코드. 예: `2026_05-4` |
| `rider_id` | 라이더 고정 ID |
| `rider_name` | 업로드 시점의 라이더명 |
| `pickup_name` | 픽업 상점명 또는 픽업지명 |
| `pickup_address` | 픽업 상세 주소 |
| `pickup_area` | 픽업 권역 또는 행정 구역 |
| `dropoff_region` | 배달 지역 상위 구분 |
| `dropoff_dong` | 배달 동 단위 정보 |
| `completed_at` | 배달 완료 시각 |
| `assigned_at` | 배정 시각 |
| `peak_time` | 피크타임 구간. 예: `Post_Lunch`, `Post_Dinner` |
| `delivery_type` | 단건/멀티배달 계열 |
| `order_no` | 주문번호 |

파생 가능한 값:

- 요일
- 시간대
- 피크타임
- 픽업지역별 완료 콜 수
- 배달동별 완료 콜 수
- 라이더별 수행 권역
- 시간대/요일별 반복 발생 권역

## 4. 픽업 / 배달 지도 표시 정책

개인정보 보호와 오해 방지를 위해 픽업지와 배달지는 표시 정책을 다르게 둔다.

| 구분 | 표시 기준 | 표시 방식 |
| --- | --- | --- |
| 픽업지 | 상세 주소 기준 표시 가능 | 상점/픽업지 마커 또는 권역 묶음 |
| 배달지 | 동 단위 중심 표시 | 동 중심점, 영역, 흐릿한 권역 표시 |
| 경로선 | 실제 주행 경로 아님 | 픽업 권역과 배달 동을 잇는 참고선 |

주의:

- 배달지를 고객 실제 위치처럼 표시하지 않는다.
- 배달 상세 주소를 지도에 찍지 않는다.
- 경로선은 실제 주행 경로가 아니라 참고선으로 표시한다.
- 지도는 코칭 참고용이며 실시간 위치 지도처럼 보이면 안 된다.

## 5. 추천 알고리즘 초안

### 1차 알고리즘

DB나 외부 지도 API 없이 업로드 데이터만으로 가능한 수준이다.

기준:

- 선택 주차
- 요일
- 시간대 또는 `peak_time`
- 픽업지역 기준 완료 콜 수

방식:

1. 선택 주차의 오더를 필터링한다.
2. 선택 요일과 시간대에 해당하는 오더를 필터링한다.
3. 픽업지역별 완료 콜 수를 집계한다.
4. 완료 콜 수 기준 TOP 5를 정렬한다.
5. 동일 시간대에 반복 발생한 지역을 우선 표시한다.

출력:

- 추천 1순위 권역
- 보조 추천 권역 2~3개
- 지난주 같은 시간대 완료 콜 수
- 추천 이유

### 2차 알고리즘

라이더 개인 경험과 복귀 판단을 반영한다.

추가 기준:

- 라이더 본인 과거 수행 경험
- 배달 완료 후 가까운 픽업 밀집 권역
- 피크타임 가중치
- 멀티배달 발생 권역 가중치

방식:

1. 전체 픽업 밀집 TOP 지역을 계산한다.
2. 본인 `rider_id`의 수행 권역을 별도로 계산한다.
3. 본인 수행 경험이 있는 권역에 기본 신뢰도를 부여한다.
4. 피크타임과 멀티배달 발생 빈도에 가중치를 준다.
5. 배달 완료 권역에서 가까운 픽업 밀집 권역을 후보로 표시한다.

주의:

- 실제 거리 계산은 Google Maps 연동 전까지 정확한 거리값으로 표현하지 않는다.
- “가까운” 표현은 좌표/거리 계산이 붙기 전에는 “복귀 후보” 또는 “인접 권역 후보” 수준으로 제한한다.

### 3차 알고리즘

DB와 지도 연동 이후의 고도화 단계다.

추가 후보:

- DB 도입 후 누적 주차 비교
- Google Maps 거리 계산
- 권역별 점수화
- 추천 신뢰도 표시
- 주차별 추세 비교

점수 예시:

```text
추천 점수 =
  픽업 완료 콜 수 점수
+ 같은 요일/시간대 반복 점수
+ 피크타임 가중치
+ 멀티배달 발생 가중치
+ 본인 수행 경험 가중치
- 거리/복귀 부담 점수
```

신뢰도 예시:

- 높음: 같은 주차/요일/시간대에 반복 픽업이 많고 본인 경험도 있음
- 보통: 전체 픽업은 많지만 본인 경험은 적음
- 참고: 데이터 수가 적어 참고용으로만 표시

## 6. UI 문구 기준

### 사용할 문구

- 선택 주차 업로드 데이터 기준
- 지난주 같은 시간대 픽업이 많았던 권역입니다
- 배차 가능성이 높았던 대기 권역 참고
- 실시간 위치가 아닙니다
- 배달지는 개인정보 보호를 위해 동 단위로 표시됩니다
- 과거 운행 데이터 기준 코칭 참고 정보입니다
- 이 권역은 배차를 보장하지 않습니다

### 사용하지 말 문구

- 실시간 배차 추천
- 현재 위치
- 관제
- 라이브 콜
- 고객 위치
- 여기 있으면 무조건 배차
- 지금 여기로 이동하세요
- 배차 확정
- 실시간 대기 포인트

문구 원칙:

- 배차 보장처럼 들리는 표현을 쓰지 않는다.
- 현재 위치를 수집하는 것처럼 보이는 표현을 쓰지 않는다.
- 고객 위치를 표시하는 것처럼 보이는 표현을 쓰지 않는다.
- 라이더가 참고해서 판단하는 코칭 정보임을 명확히 한다.

## 7. 라이더 화면 UI 초안

카드명:

- 어디서 대기할까?

표시 항목:

| 항목 | 예시 |
| --- | --- |
| 추천 1순위 권역 | 용전동 픽업권역 |
| 추천 이유 | 지난주 같은 시간대 픽업이 가장 많았던 권역입니다. |
| 시간대 | Post_Dinner |
| 요일 | 금요일 |
| 지난주 기준 완료 콜 수 | 18건 |
| 보조 추천 권역 | 홍도동, 가양동, 성남동 |
| 안내 문구 | 선택 주차 업로드 데이터 기준입니다. 실시간 위치가 아닙니다. |

라이더 카드 구성 예:

```text
어디서 대기할까?

추천 1순위
용전동 픽업권역

지난주 금요일 Post_Dinner에 픽업이 많았던 권역입니다.

보조 추천
홍도동 · 가양동 · 성남동

안내
선택 주차 업로드 데이터 기준입니다.
배차를 보장하지 않으며 실시간 위치가 아닙니다.
```

라이더 화면 배치 후보:

- 내지도 화면 상단 카드
- 내현황의 오늘의 페이스 체크 아래 보조 카드
- 내코칭의 실천 미션 하단 참고 카드

1차 적용 우선 후보:

- 내지도 화면에 “추천권역 카드” 추가
- 지도 실제 연동 전에는 리스트/카드 형태로 먼저 제공

## 8. 관리자 화면 UI 초안

카드명:

- 시간대별 픽업 밀집 분석
- 라이더별 배차 포지션 분석

표시 항목:

| 항목 | 설명 |
| --- | --- |
| 주차 선택 | 분석할 업로드 주차 |
| 요일 선택 | 월~일 또는 전체 |
| 시간대 선택 | Breakfast, Lunch_Peak, Post_Lunch, Dinner_Peak, Post_Dinner |
| 라이더 선택 | 전체 또는 특정 라이더 |
| 픽업지역 TOP 5 | 픽업 완료 콜 수 기준 |
| 배달동 TOP 5 | 동 단위 집계 |
| 추천 대기권역 | 선택 조건 기준 추천 |
| 코칭 메모 | 관리자 코칭 메시지 작성 참고 |

관리자 카드 구성 예:

```text
시간대별 픽업 밀집 분석

주차: 2026_05-4
요일: 금요일
시간대: Post_Dinner

픽업지역 TOP 5
1. 용전동 18건
2. 가양동 14건
3. 홍도동 11건

추천 대기권역
용전동 → 가양동 권역 참고

코칭 메모
금요일 저녁에는 용전동 권역 대기 전략을 안내할 수 있습니다.
```

관리자 화면 주의:

- 라이더 선택은 관리자 화면에만 둔다.
- 라이더 화면에는 타 라이더 선택 UI를 만들지 않는다.
- 배달동은 동 단위로만 표시한다.

## 9. 지도 연동 단계

| 단계 | 내용 | 구현 범위 |
| --- | --- | --- |
| 1단계 | 문서화 | 현재 단계. 기능 구현 없음 |
| 2단계 | 데이터 타입 추가 | 픽업/배달 권역 분석용 타입과 순수 집계 함수 설계 |
| 3단계 | 내지도 추천권역 카드 | 지도 API 없이 카드/리스트로 추천권역 표시 |
| 4단계 | 관리자 픽업 밀집 분석 | 관리자 화면에 TOP 지역/요일/시간대 분석 추가 |
| 5단계 | 좌표 캐시 설계 | 픽업 주소 좌표 변환 결과 캐시 구조 설계 |
| 6단계 | Google Maps 실제 연동 | 공식 API 검토 후 지도 표시/거리 계산 도입 |

Google Maps 실제 연동 전 확인:

- API 비용
- 주소 지오코딩 제한
- 키 보안과 환경변수 관리
- 좌표 캐시 저장 위치
- 배달지 동 단위 표시 정책
- 실시간 위치처럼 보이지 않는 UI 문구

## 10. 개인정보 / 운영 주의사항

- 배달지는 동 단위만 사용한다.
- 고객 주소 추정을 금지한다.
- 원천 엑셀 파일 저장을 금지한다.
- 실시간 위치 추적을 금지한다.
- 라이더는 본인 데이터만 조회한다.
- 관리자는 전체 권역 분석 가능하되 원천 개인정보를 노출하지 않는다.
- 지도에 표시되는 정보는 과거 업로드 데이터 기반의 코칭 참고 정보다.
- 배차 가능성이 높았던 권역이라는 표현은 가능하지만 배차 보장 표현은 금지한다.
- 정산 데이터와 직접 연동하지 않는다.
- 현재 위치/GPS 권한 요청을 하지 않는다.

운영 전 결정할 항목:

| 항목 | 결정 필요 |

### 신뢰도 레벨 기능

신뢰도 계산 기준:

- **보통 (medium)**: 완료 콜이 3~5건 → "신뢰도 보통"
- **참고용 (low)**: 완료 콜이 1~2건 → "참고용"

구현 함수:
- `getConfidenceLevel(completedCount)`: 콜 수 기반 신뢰도 레벨 계산
- `getConfidenceLevelLabel(level)`: 한국어 레벨 라벨 반환
- `getConfidenceLevelDescription(level)`: 신뢰도 설명 문구 반환

UI 개선:
- 라이더 카드: 각 추천 권역별 신뢰도 배지 표시 (색상 코딩: 높음=녹색, 보통=주황색, 참고=회색)
- 관리자 카드: 각 TOP 권역별 신뢰도 배지 표시
- 라이더 추천 문구 개선:
  - 높음: "지난 업로드 주차 기준, 내가 픽업을 많이 수행한 권역입니다."
  - 보통/참고: "선택 주차에 내가 픽업을 많이 수행한 권역에서 참고용 데이터를 완료 최 수가 많지 않기 때문에 참고용으로만 확인하세요."

테스트 추가:
- 신뢰도 레벨 계산 검증 (1-2=low, 3-5=medium, 6+=high)
- 한국어 레벨 라벨 검증
- 설명 문구 검증

## 31단계 필터 설계 보강

31단계는 기능 개발이 아니라 다음 확장을 위한 설계 문서 보강 단계다.

이번 단계에서는 Google Maps/API/GPS/DB/Supabase, 외부 API, 위치 권한, 실시간 데이터 연결을 추가하지 않는다. 기존 업로드/검수/코칭/정산앱 링크/라이더 화면 로직도 변경하지 않는다.

### 31-1. 필터 도입 목적

전체 주차 기준 TOP 권역만으로는 실제 운행 전략을 세우기에 정보가 부족하다. 같은 주차 안에서도 요일, 시간대, 피크타임에 따라 픽업이 많이 발생하는 권역이 달라질 수 있기 때문이다.

라이더에게 중요한 질문은 단순히 "어느 권역이 전체적으로 많았는가"가 아니라 "선택한 요일/시간대에는 어디서 대기하면 참고할 만한가"이다. 따라서 라이더 화면은 최신 업로드 주차를 기본으로 하되 요일/시간대 기준으로 본인 수행 권역을 좁혀 볼 수 있어야 한다.

관리자는 시간대/요일별 픽업 밀집 권역을 보고 코칭에 활용할 수 있다. 예를 들어 특정 라이더가 특정 시간대에 강한 권역과 약한 권역을 비교하고, 다음 주 코칭 메시지에 참고 권역을 짧게 안내할 수 있다.

### 31-2. 라이더용 필터 방향

기본값은 최신 업로드 주차 전체로 둔다. 라이더 화면에서는 필터가 많아지면 복잡해지므로 1차 화면에는 1~2개 필터만 먼저 제공하고, 이후 단계에서 확장한다.

라이더용 필터 후보:

- 최신 업로드 주차 전체
- 오늘 요일 기준 보기
- 선택 요일 기준 보기
- 시간대 기준 보기
- 피크타임 기준 보기
- 내 과거 수행 권역 기준 보기

라이더 화면 문구 예:

- 선택 주차 업로드 데이터 기준입니다.
- 선택한 요일/시간대에 내가 픽업을 많이 수행한 권역입니다.
- 실시간 위치가 아닙니다.
- 배차를 보장하지 않으며 참고용 권역입니다.

라이더 데이터 기준:

- 라이더 화면은 로그인한 사용자의 `rider_id` 데이터만 사용한다.
- 다른 라이더의 권역을 fallback 추천으로 섞지 않는다.
- 데이터가 부족하면 빈 상태를 보여주고, 전체 라이더 데이터로 대체하지 않는다.

### 31-3. 관리자용 필터 방향

관리자 화면은 운영/코칭 참고용 분석이 목적이므로 라이더 화면보다 많은 필터를 제공할 수 있다. 다만 모바일 폭에서는 한 번에 모든 필터를 노출하지 않고 접이식 필터 패널을 우선 검토한다.

관리자용 필터 후보:

- 주차 선택
- 요일 선택
- 시간대 선택
- 피크타임 선택
- 라이더 선택
- 배달타입 선택
- 전체/라이더별 비교

관리자 화면 문구 예:

- 선택 주차 업로드 데이터 기준입니다.
- 선택한 요일/시간대의 픽업 밀집 권역입니다.
- 실시간 위치 또는 현재 배차 현황이 아닙니다.

관리자 데이터 기준:

- 관리자는 전체 데이터 조회가 가능하다.
- 라이더 선택 필터는 관리자 화면에서만 제공한다.
- 전체/라이더별 비교는 코칭 참고 목적이며 배차 보장 문구로 표현하지 않는다.

### 31-4. 필터 우선순위

1차:
- 시간대 필터

2차:
- 요일 필터

3차:
- 피크타임 필터

4차:
- 라이더 선택 필터

5차:
- 주차 선택 필터

6차:
- 배달타입 필터

우선순위 기준:

- 시간대 필터는 라이더가 바로 행동 참고로 이해하기 쉬워 가장 먼저 도입한다.
- 요일 필터는 주간 패턴을 좁히는 데 유용하지만 데이터 수가 줄어들 수 있어 2차로 둔다.
- 피크타임 필터는 기존 `peakTime` 값의 품질과 표준화 상태를 확인한 뒤 도입한다.
- 라이더 선택 필터는 관리자 화면 전용으로 둔다.
- 주차 선택 필터는 누적 주차 데이터 저장 구조가 안정화된 뒤 확장한다.
- 배달타입 필터는 표본 수가 작아질 수 있으므로 마지막에 검토한다.

### 31-5. 모바일 UX 기준

- 필터는 칩 버튼 방식을 우선한다.
- 기본값은 "전체"로 둔다.
- 너무 많은 필터를 한 화면에 노출하지 않는다.
- 라이더 화면은 1~2개 필터만 먼저 제공한다.
- 관리자 화면은 접이식 필터 패널을 사용할 수 있다.
- 360px 모바일 폭에서 버튼이 겹치지 않게 구성한다.
- 칩은 줄바꿈 가능한 가로 스크롤 또는 2열 배치 중 화면 밀도에 맞는 방식을 선택한다.
- 선택된 칩은 색상과 굵기로만 구분하지 않고 텍스트 상태도 함께 명확히 한다.

## 32단계 시간대 필터 유틸 보강

32단계에서는 UI를 변경하지 않고 배차 포지션 추천 순수 유틸의 시간대 필터 기준을 보강했다.

보강 범위:

- 시간대 필터 값에 `all`을 추가해 기본 전체 조회 기준을 명확히 했다.
- `timeSlot`이 없거나 `all`이면 전체 오더를 포함한다.
- 특정 `timeSlot`이 있으면 해당 시간대 오더만 집계한다.
- `riderId`와 `timeSlot` 필터가 동시에 적용될 수 있게 유지했다.
- `peakTime` 값이 해석 가능하면 `completedAt`/`assignedAt`보다 먼저 사용한다.

시간대 기준:

| 시간대 | 기준 |
| --- | --- |
| `morning` | 06:00~10:59 |
| `lunch_peak` | 11:00~12:59 |
| `post_lunch` | 13:00~16:59 |
| `dinner_peak` | 17:00~19:59 |
| `post_dinner` | 20:00~23:59 |
| `late_night` | 00:00~05:59 |
| `unknown` | 시간값 없음 또는 파싱 불가 |

피크타임 우선 매핑:

- `Post_Lunch` -> `post_lunch`
- `Post_Dinner` -> `post_dinner`
- `Lunch`, `Lunch_Peak` -> `lunch_peak`
- `Dinner`, `Dinner_Peak` -> `dinner_peak`
