# 베타 피드백 반영 우선순위 계획

## 목적

라이더 코칭센터 Ver.2 1차 베타 테스트에서 들어오는 피드백을 어떤 기준으로 분류하고 어떤 순서로 반영할지 정리한다.

이번 문서는 베타 운영 문서다. 기능 코드, DB/Supabase 연결, 외부 API, 문자, GPS, 실시간 콜, 실제 음악 재생, Vercel/GitHub 배포 설정은 추가하거나 변경하지 않는다. 실제 개인정보, 원천 엑셀 파일, 정산 원본 파일도 포함하지 않는다.

## A. 피드백 처리 원칙

- 모든 피드백을 바로 기능 개발로 연결하지 않는다.
- 먼저 오류, 문구 개선, 화면/디자인 개선, 기능 추가 요청, 데이터/엑셀 관련, 정산앱 관련, 로그인/권한 관련으로 분류한다.
- 우선순위는 긴급, 높음, 보통, 낮음으로 나눈다.
- 라이더 개인정보, 권한 노출, 로그인 문제는 최우선 처리한다.
- 라이더가 타 라이더 데이터를 보는 문제는 베타 공개를 중단할 수 있는 항목으로 본다.
- 기능 추가 요청은 1차 베타 즉시 수정이 아니라 2차 개발 후보로 따로 분리한다.
- DB/Supabase 관련 요청은 운영 전 별도 의사결정 항목으로 분리한다.
- 정산 계산, 지급, 확정, 정산 엑셀 파싱 요청은 코칭센터가 아니라 기존 정산앱 개선 요청인지 먼저 구분한다.
- 원천 엑셀 파일, 실제 개인정보, 정산 원본 파일은 피드백 자료에 첨부하지 않는다.

## B. 우선순위 기준

### 긴급

발견 즉시 원인을 확인하고, 필요하면 베타 URL 공유나 테스트 진행을 멈춘다.

| 유형 | 예시 | 기본 조치 |
| --- | --- | --- |
| 로그인 불가 | 모든 계정 또는 특정 역할 계정이 로그인되지 않음 | 즉시 재현, 원인 확인 |
| 화면 진입 불가 | 관리자/라이더 화면이 열리지 않음 | 공개 보류 검토 |
| 데이터 권한 노출 | 라이더가 타 라이더 데이터를 볼 수 있음 | 즉시 수정 전까지 공개 중단 |
| 관리자 기능 노출 | 관리자 기능이 라이더에게 보임 | 즉시 수정 |
| 잘못된 외부 링크 | 정산앱 링크가 엉뚱한 주소로 연결됨 | 링크 차단 또는 수정 |
| 업로드 후 화면 전체 깨짐 | 반영 후 주요 화면 사용 불가 | 업로드/반영 흐름 점검 |

### 높음

베타 진행은 가능할 수 있지만, 1차 베타 종료 전 원인 확인과 수정 계획이 필요하다.

| 유형 | 예시 | 기본 조치 |
| --- | --- | --- |
| 주요 KPI 수치 오류 | 완료 콜, 운영 라이더, 검수 이슈 수가 명백히 틀림 | 데이터 변환/집계 검토 |
| 엑셀 흐름 오류 | 업로드/검수/반영 흐름이 기대와 다름 | 파일명/시트/반영 상태 확인 |
| 페이스 계산 오류 | 목표까지 필요한 추가 콜 수가 잘못 계산됨 | 계산 로직 점검 |
| 코칭 노출 오류 | 노출 OFF 메시지가 라이더에게 보임 | 권한/노출 상태 점검 |
| 모바일 메뉴 사용 불가 | 하단 메뉴가 눌리지 않거나 가림 | 모바일 레이아웃 수정 |
| 정산앱 링크 실패 | 버튼 클릭 시 정산앱이 열리지 않음 | URL/새 탭 동작 확인 |

### 보통

사용은 가능하지만 베타 경험을 개선하기 위해 정리할 항목이다.

| 유형 | 예시 | 기본 조치 |
| --- | --- | --- |
| 문구 혼란 | 최신 업로드 주차, 샘플 데이터, 임시 저장 안내가 헷갈림 | 문구 수정 후보 |
| 버튼 위치 불편 | 주요 버튼을 찾기 어려움 | UX 개선 후보 |
| 화면이 김 | 카드가 길어 한 번에 이해하기 어려움 | 우선순위/접기 개선 후보 |
| 일부 카드 깨짐 | 특정 휴대폰에서 일부 카드만 어색함 | 화면별 polish 후보 |
| 이슈 설명 부족 | 검수 이슈 의미가 부족함 | 도움 문구 보강 |
| 안내 문구 개선 | 정산앱/코칭센터 역할 안내 개선 요청 | 문구 개선 후보 |

### 낮음

1차 베타 즉시 수정 대상은 아니며, 추후 디자인/기능 후보로 보관한다.

| 유형 | 예시 | 기본 조치 |
| --- | --- | --- |
| 색상 선호 | 버튼 색상이나 배지 색상 변경 요청 | 디자인 후보 |
| 아이콘 변경 | 메뉴 아이콘 변경 요청 | 디자인 후보 |
| 추가 차트 요청 | 새로운 그래프/통계 요청 | 2차 후보 |
| 개인화 기능 요청 | 개인 목표 추천, 개인별 설정 확대 | 2차 후보 |
| 음악 기능 확장 요청 | 실제 재생, 플레이리스트 연결 요청 | 베타 제외, 2차 후보 |
| 지도 고도화 요청 | 히트맵, 상세 지도, 경로 분석 요청 | 2차 후보 |

## E. 1차 베타에서 즉시 처리할 항목

아래 항목은 1차 베타에서 즉시 확인하고 가능한 한 우선 처리한다.

- 로그인/권한 문제
- 관리자/라이더 화면 진입 불가
- 라이더 본인 외 데이터 노출 문제
- 관리자 기능이 라이더에게 노출되는 문제
- 업로드/반영 불가
- 정산앱 링크 오류
- 모바일 화면의 심각한 깨짐
- 잘못된 파일 선택 후 기존 정상 데이터가 깨지는 문제

즉시 처리 판단 기준:

- 베타 참여자가 앱을 계속 테스트할 수 없는가?
- 개인정보 또는 권한 문제가 생기는가?
- 주요 데이터 흐름이 잘못 보이는가?
- 정산앱 링크가 잘못된 곳으로 이동하는가?

하나라도 해당하면 긴급 또는 높음으로 분류한다.

## F. 2차 개발 후보로 분리할 항목

아래 요청은 중요하더라도 1차 베타 즉시 수정 대상이 아니라 2차 개발 후보로 분리한다.

| 후보 항목 | 분리 이유 |
| --- | --- |
| DB 저장 | 운영 저장 구조와 개인정보 정책 결정 필요 |
| Supabase 로그인 | Auth, RLS, 계정 정책, 배포 환경 검토 필요 |
| 개인 계정 발급 | 실제 계정 운영 정책과 승인 절차 필요 |
| 정산 데이터 직접 연동 | 정산앱 책임 영역과 권한 통합 검토 필요 |
| 지도 히트맵 | 위치/지역 데이터 처리 범위 검토 필요 |
| 음악 실제 재생 | 안전 문구, 외부 링크, 재생 권한 검토 필요 |
| 알림/문자 발송 | 개인정보, 발송 비용, 수신 동의 검토 필요 |
| 배민플러스 등 타 플랫폼 확장 | 현재 앱은 쿠팡플러스 전용 기준 |

DB/Supabase 요청 처리 원칙:

- “새로고침 후 데이터 유지가 필요하다”는 피드백은 DB 도입 검토 항목으로 분리한다.
- “개인 계정이 필요하다”는 피드백은 로그인/계정 정책 항목으로 분리한다.
- “라이더별 데이터 접근을 더 강하게 막아야 한다”는 피드백은 RLS/권한 정책 항목으로 분리한다.
- 실제 Supabase 설치, 패키지 추가, 연결 구현은 별도 승인 후 진행한다.

## G. 베타 종료 후 정리 절차

1. 중복 피드백 묶기
   - 같은 화면, 같은 원인, 같은 문구 요청은 하나로 합친다.

2. 긴급/높음 먼저 처리
   - 로그인, 권한, 데이터 노출, 업로드/반영, 정산앱 링크 문제를 먼저 확인한다.

3. 보통/낮음은 UX 개선 목록으로 분리
   - 문구, 버튼 위치, 카드 길이, 색상, 아이콘 요청은 UX 개선 후보로 묶는다.

4. 기능 추가 요청은 2차 개발 후보로 분리
   - 새 기능 요청은 현재 베타 안정화와 분리한다.

5. DB/Supabase 요청은 별도 의사결정 회의로 분리
   - 비용, 개인정보, 권한, RLS, 운영 리스크를 별도로 판단한다.

6. 최종 반영 목록과 보류 목록 작성
   - 반영 완료, 수정 예정, 보류, 반려 항목을 명확히 남긴다.

## 최종 원칙

1. 베타 피드백은 안정성, 권한, 데이터 정확성을 먼저 본다.
2. 사용성 개선은 중요하지만 개인정보와 권한 문제보다 앞서지 않는다.
3. 기능 추가 요청은 2차 개발 후보로 분리한다.
4. DB/Supabase 요청은 별도 운영 판단 후 진행한다.
5. 정산앱 관련 요청은 코칭센터 기능인지 정산앱 개선 요청인지 먼저 구분한다.
