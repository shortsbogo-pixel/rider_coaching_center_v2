# 저장 구조 설계

## 목표

현재 앱은 `latestUploadedWeekData`를 브라우저 세션 상태로만 유지한다. 4단계에서는 실제 DB, Supabase, 외부 API를 연결하지 않고, 향후 베타/정식 운영에서 저장소로 교체할 수 있는 저장 어댑터 구조만 정의한다.

원천 엑셀 파일 자체는 저장하지 않는다. 저장 대상은 브라우저에서 파싱, 검수, 정규화, 집계가 끝난 데이터다.

## 저장 대상

코드 기준 저장 레코드는 `StoredWeekDataRecord`이며, 운영 DB 컬럼으로 전환할 때는 아래 의미를 유지한다.

| 운영 필드 | 코드 필드 | 설명 |
| --- | --- | --- |
| `week_code` | `weekCode` | 내부 주차 코드. 예: `2026_05-4` |
| `week_label` | `weekLabel` | 화면 표시 주차. 예: `5월4주차` |
| `uploaded_at` | `uploadedAt` | 정규화 데이터 저장 시각 |
| `source_file_name` | `sourceFileName` | 원천 파일명 문자열만 저장 |
| `inspection_summary` | `inspectionSummary` | 총행수, 유효 오더 후보, 확인 필요 행, 이슈 건수, 주요 이슈 유형 |
| `normalized_orders` | `normalizedOrders` | 정규화된 오더 이력 |
| `rider_weekly_summaries` | `riderWeeklySummaries` | 라이더별 주간 집계 |
| `coaching_messages` | `coachingMessages` | 자동 초안, 관리자 수정 메시지, 내부 메모 |
| `admin_visibility_settings` | `adminVisibilitySettings` | 라이더별 노출 ON/OFF 상태 |

## 저장하지 않을 대상

- 원천 엑셀 파일 바이너리
- 브라우저 `File` 객체 또는 ArrayBuffer
- 파싱 전 원본 행 전체
- 불필요한 개인정보 원본
- 실시간 위치 정보
- 실시간 배달 상태 또는 관제 데이터

## 저장 방식 후보

| 단계 | 방식 | 장점 | 위험 | 전환 시점 |
| --- | --- | --- | --- | --- |
| MVP/개발 | in-memory only | 구현이 단순하고 민감 데이터가 남지 않음 | 새로고침/서버 재시작 시 데이터 소실 | 현재 단계 |
| 베타 | server JSON store 또는 Manus 런타임 저장 | DB 설계 전 파일 단위 운영 검증 가능 | 동시 업로드 충돌, 권한/감사 로그 부족, 백업 정책 필요 | 제한된 베타 운영에서 주차별 업로드 지속성이 필요할 때 |
| 정식 운영 | Supabase/PostgreSQL | RLS, 인덱스, 감사 로그, 백업, 이력 관리에 적합 | 스키마/권한 설계 필요, 개인정보 보안 책임 증가 | 외부 라이더 접속과 운영 데이터 보존이 필요할 때 |

## 코드 구조

현재 추가된 저장 어댑터:

- `WeekDataRepository`
- `loadLatestWeekData`
- `saveLatestWeekData`
- `clearLatestWeekData`
- `InMemoryWeekDataRepository`

현재 앱 화면에는 저장소를 연결하지 않는다. UI는 기존처럼 브라우저 상태를 사용한다. 이후 저장소 연결 시 화면은 `WeekDataRepository`를 통해 최신 주차 데이터를 불러오도록 교체한다.

## xlsx 운영 전 보안 판단 항목

현재 앱은 브라우저에서 선택한 `.xlsx` 파일을 파싱한다. 운영 전에는 `xlsx` 패키지에 대해 외부 감사 도구 또는 보안 자문 결과로 보고되는 취약점 3건을 각각 아래 기준으로 판단한다.

1. `xlsx` 취약점 1
   - 실제 운영 업로드 경로에서 도달 가능한지 확인
   - 악성 파일 파싱 시 브라우저 멈춤, 메모리 사용량 증가, 데이터 변조 가능성 평가
   - 완화책: 파일 크기 제한, 업로드 주체 제한, 파싱 워커 분리, 대체 라이브러리 검토

2. `xlsx` 취약점 2
   - 서버 저장소 연결 후 서버 측 파싱으로 이동할 가능성이 있는지 확인
   - 서버 측 파싱 시 권한 격리, 타임아웃, 샌드박스 필요성 평가
   - 완화책: 서버 파싱 금지 유지 또는 격리된 파싱 런타임 사용

3. `xlsx` 취약점 3
   - 셀 텍스트를 화면에 표시할 때 스크립트/수식/HTML이 실행되지 않는지 확인
   - 관리자 내부 메모와 코칭 메시지로 전파될 수 있는 문자열을 검토
   - 완화책: React 기본 escaping 유지, HTML 삽입 금지, 파일 확장자/시트/헤더 검증 유지

운영 전에는 보안 감사 실행 자체가 프로젝트 의존성 정보를 외부 서비스로 전송할 수 있으므로, 실행 주체와 승인 절차를 먼저 정한다.
