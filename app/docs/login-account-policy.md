# Login and Account Policy

## 목적

라이더 코칭센터 Ver.2의 1차 베타 이후 로그인, 계정, 권한 운영 기준을 정리한다.

이번 문서는 운영 정책 문서이며, 실제 인증 구현, DB/Supabase 연결, 외부 API 연결은 하지 않는다. 실제 개인정보나 운영 계정 정보도 포함하지 않는다.

## 현재 베타 로그인 구조

현재 로그인은 실제 인증 시스템이 아니라 베타 화면 흐름을 확인하기 위한 테스트 계정 방식이다.

| 역할 | 아이디 | 비밀번호 | 상태 | 목적 |
| --- | --- | --- | --- | --- |
| 관리자 | `admin` | `admin1234` | active | 관리자 대시보드, 업로드, 검수, 코칭, 더보기 확인 |
| 라이더 | `rider1` | `rider1234` | active | 라이더 본인 화면과 오늘의 페이스 체크 확인 |
| 승인대기 | `pending` | `pending1234` | pending | 승인 전 접근 제한 확인 |

테스트 계정은 베타 검수용이다. 운영 공개 전에는 폐기하거나 안내를 숨겨야 한다.

현재 로그인 화면의 테스트 계정 안내는 `src/lib/beta.ts`의 `showTestAccounts` 플래그로 숨길 수 있다.

```ts
export const betaMode = true;
export const showTestAccounts = betaMode;
```

운영 공개 전에는 `showTestAccounts`를 `false`로 바꾸고, 로그인 화면에서 테스트 계정 카드가 보이지 않는지 확인한다.

## 베타 이후 권장 로그인 정책

- 공용 테스트 계정을 종료한다.
- 관리자 개인 계정을 발급한다.
- 라이더 개인 계정을 발급한다.
- 가입 직후 또는 검토 중 계정은 `pending` 상태로 둔다.
- 라이더별 `rider_id`를 고정 매핑한다.
- 라이더는 본인 `rider_id` 데이터만 조회한다.
- 관리자는 전체 주차와 전체 라이더 데이터를 조회할 수 있다.
- 승인대기 계정은 관리자/라이더 데이터 화면에 접근할 수 없다.
- 계정 발급, 비활성화, 권한 변경 이력을 운영 로그 또는 감사 로그로 남긴다.

## 역할별 권한 정책

### `admin`

관리자는 운영 관리 권한을 가진다.

| 권한 | 허용 여부 |
| --- | --- |
| 엑셀 업로드 | 허용 |
| 업로드 미리보기 생성 | 허용 |
| 이번 주차 데이터 반영 | 허용 |
| 검수 결과 조회 | 허용 |
| 전체 대시보드 조회 | 허용 |
| 전체 라이더 목록 조회 | 허용 |
| 코칭 메시지 관리 | 허용 |
| 라이더 노출 ON/OFF 변경 | 허용 |
| 내부 메모 작성 | 허용 |
| 페이스 체크 설정 변경 | 허용 |
| 운영 로그 조회 | 허용 |
| 라이더 화면 본인 입력 대리 작성 | 기본 비허용, 별도 운영 정책 필요 |

### `rider`

라이더는 본인 데이터 확인 권한만 가진다.

| 권한 | 허용 여부 |
| --- | --- |
| 본인 내현황 조회 | 허용 |
| 본인 내오더 조회 | 허용 |
| 본인 내지도 조회 | 허용 |
| 본인 내코칭 조회 | 허용 |
| 오늘의 페이스 체크 수기 입력 | 허용 |
| 정산앱 링크 이동 | 허용 |
| 관리자 화면 접근 | 비허용 |
| 전체 라이더 목록 조회 | 비허용 |
| 타 라이더 데이터 조회 | 비허용 |
| 업로드/검수/코칭 관리 | 비허용 |

라이더 화면에는 타 라이더 선택 UI를 두지 않는다.

### `pending`

승인대기 계정은 데이터 접근 권한을 갖지 않는다.

| 권한 | 허용 여부 |
| --- | --- |
| 승인대기 안내 조회 | 허용 |
| 관리자 화면 접근 | 비허용 |
| 라이더 데이터 화면 접근 | 비허용 |
| 본인 오더/코칭 조회 | 비허용 |
| 오늘의 페이스 체크 입력 | 비허용 |
| 정산앱 링크 접근 | 운영 정책으로 결정, 기본은 비허용 권장 |

## `rider_id` 매핑 정책

`rider_id`는 라이더 계정과 주차별 운행 데이터를 연결하는 기준이다.

운영 기준:

- 라이더 계정 1개는 하나의 고정 `rider_id`를 가진다.
- 주차별 업로드 데이터의 라이더명은 `rider_id`로 매핑되어야 한다.
- 이름이 변경되어도 `rider_id`는 유지한다.
- 탈퇴, 이탈, 비활성 계정도 과거 데이터 추적을 위해 기존 `rider_id`를 재사용하지 않는다.
- 라이더가 로그인했을 때는 본인 `rider_id` 기준 데이터만 조회한다.

## 정산앱과의 계정 정책

초기 운영에서는 정산앱과 코칭센터를 각각 로그인하는 방식을 권장한다.

- 자동 로그인 또는 SSO는 이번 베타 범위에서 제외한다.
- 코칭센터의 정산앱 링크는 이동 링크이며 권한 통합이 아니다.
- 정산앱의 인증, 권한, 지급 상태, 정산 확정은 정산앱에서 처리한다.
- 향후 DB 도입 후 공통 `rider_id` 기준 연동을 검토할 수 있다.
- 공통 계정 또는 SSO를 도입하려면 개인정보 처리, 권한 범위, 로그 정책을 별도로 승인해야 한다.

## 보안/운영 권장사항

- 관리자 계정 수를 최소화한다.
- 운영 공개 전 테스트 계정 안내를 숨긴다.
- 공용 비밀번호 공유를 금지한다.
- 초기 비밀번호 발급 후 변경 정책을 만든다.
- 비밀번호 재설정 흐름을 정한다.
- 퇴사/이탈 라이더 계정 비활성화 정책을 만든다.
- 관리자 권한 변경 기록을 남긴다.
- 업로드, 반영, 코칭 메시지 변경, 페이스 설정 변경은 운영 로그에 남긴다.
- 원천 엑셀 파일 미저장 원칙을 유지한다.
- 개인정보는 최소 범위만 저장한다.
- 실제 전화번호, 계좌번호, 주민등록번호, 상세 주소는 기본 저장 대상에서 제외한다.
- 정산 최종 파일은 코칭센터 업로드 대상에서 계속 제외한다.

## 향후 Supabase/Auth 도입 시 고려사항

Supabase/Auth 도입은 별도 승인 후 진행한다.

검토 항목:

- `users` 테이블과 Supabase Auth 사용자 ID 매핑
- `rider_id` 고정 매핑
- `role` 기반 접근 제어
- `account_status` 기반 승인대기 분리
- Row Level Security 정책
- 관리자 전체 조회 정책
- 라이더 본인 데이터 조회 정책
- 승인대기 데이터 접근 차단 정책
- 비밀번호 재설정 흐름
- 관리자 권한 변경 감사 로그
- 업로드/반영/설정 변경 운영 로그
- 테스트 계정 제거 또는 seed 데이터 분리

RLS 개념 초안:

| 테이블 | 관리자 | 라이더 | 승인대기 |
| --- | --- | --- | --- |
| `users` | 전체 관리 | 본인 프로필 일부 조회 | 본인 상태만 조회 |
| `upload_weeks` | 전체 조회/관리 | 최신 반영 주차 일부 조회 | 비허용 |
| `normalized_orders` | 전체 조회 | 본인 `rider_id`만 조회 | 비허용 |
| `rider_weekly_summaries` | 전체 조회 | 본인 `rider_id`만 조회 | 비허용 |
| `coaching_messages` | 전체 관리 | 본인 노출 메시지만 조회 | 비허용 |
| `pace_check_entries` | 필요 시 전체 조회 | 본인 입력 생성/수정/조회 | 비허용 |
| `operation_logs` | 전체 조회 | 비허용 | 비허용 |

| --- | --- | --- |

| 테스트 계정 안내 | `showTestAccounts`를 운영 공개 전 `false`로 변경 |  |

| rider_id | 라이더별 고정 `rider_id` 매핑표 확정 |  |
| 권한 검증 | 라이더가 본인 외 데이터를 볼 수 없는지 확인 |  |

| 테스트 데이터 | 운영 DB에서 테스트 seed와 실제 데이터를 분리 |  |

## 운영 전 최종 원칙

1. 라이더는 본인 데이터만 본다.
2. 관리자는 전체 데이터를 보되 변경 이력을 남긴다.
3. 승인대기 계정은 데이터 화면에 접근하지 못한다.
4. 테스트 계정은 운영 공개 전에 숨기거나 제거한다.
5. 정산앱은 별도 시스템이며 코칭센터와 권한이 통합된 것이 아니다.
6. 원천 엑셀 파일과 불필요한 개인정보는 저장하지 않는다.
