# 로그인 / 계정 / 권한 정책

## 목적

라이더 코칭센터 Ver.2의 1차 베타 이후 로그인, 계정, 권한 운영 기준을 정리한다.

이 문서는 운영 정책 문서다. 실제 인증 구현, DB/Supabase 연결, 외부 API 연결, Vercel/GitHub 배포 설정 변경은 하지 않는다. 실제 개인정보나 운영 계정 정보도 포함하지 않는다.

현재 앱은 쿠팡플러스 전용 코칭 도구이며, 정산앱은 별도 시스템이다. 코칭센터의 정산앱 버튼은 이동 링크일 뿐 계정 통합이나 권한 통합이 아니다.

## 현재 베타 로그인 구조

현재 로그인은 실제 인증 시스템이 아니라 베타 흐름 검수용 테스트 계정 방식이다.

| 구분 | ID | 비밀번호 | role | account_status | 목적 |
| --- | --- | --- | --- | --- | --- |
| 관리자 | `admin` | `admin1234` | `admin` | `active` | 관리자 대시보드, 업로드, 검수, 코칭, 더보기 확인 |
| 라이더 | `rider1` | `rider1234` | `rider` | `active` | 본인 라이더 화면과 오늘의 페이스 체크 확인 |
| 승인대기 | `pending` | `pending1234` | `rider` | `pending` | 승인대기 계정의 데이터 접근 차단 확인 |

운영 전 처리 원칙:

- 테스트 계정은 베타 검수용이다.
- 운영 공개 전에는 폐기하거나 로그인 화면 안내에서 숨긴다.
- 테스트 계정 비밀번호를 실제 운영 계정에 재사용하지 않는다.
- 베타 계정은 실제 라이더 개인정보와 연결하지 않는다.

현재 로그인 화면의 테스트 계정 안내는 `showTestAccounts` 플래그로 숨길 수 있다.

```ts
export const betaMode = true;
export const showTestAccounts = betaMode;
```

운영 공개 전에는 `showTestAccounts`를 `false`로 바꾸거나, 환경별 설정으로 분리해 로그인 화면에서 테스트 계정 카드가 보이지 않도록 한다.

## 베타 이후 권장 로그인 정책

베타 이후에는 공용 테스트 계정 구조를 종료하고 개인 계정 구조로 전환한다.

권장 방향:

- 공용 테스트 계정 종료
- 관리자 개인 계정 발급
- 라이더 개인 계정 발급
- 승인대기 상태 관리
- 라이더별 `rider_id` 고정 매핑
- 라이더는 본인 데이터만 조회
- 관리자는 전체 데이터 접근 가능
- 승인대기 계정은 데이터 접근 불가

운영 계정 생성 기준:

| 계정 유형 | 발급 기준 | 활성화 기준 | 비활성화 기준 |
| --- | --- | --- | --- |
| 관리자 | 운영 책임자 승인 후 발급 | 업무상 관리자 기능 필요 | 권한 회수, 퇴사, 역할 변경 |
| 라이더 | 실제 라이더 식별 후 발급 | `rider_id` 매핑 완료 | 이탈, 계약 종료, 장기 미사용 |
| 승인대기 | 가입 또는 사전 등록 직후 | 관리자 승인 및 `rider_id` 매핑 완료 | 반려 또는 미확인 장기 보류 |

## 역할별 권한 정책

### admin

관리자는 운영 관리 목적의 전체 데이터 접근 권한을 가진다.

| 기능 | 권한 |
| --- | --- |
| 엑셀 업로드 | 가능 |
| 업로드 미리보기 확인 | 가능 |
| 이번 주차 데이터 반영 | 가능 |
| 검수 결과 확인 | 가능 |
| 전체 대시보드 조회 | 가능 |
| 전체 라이더 목록/집계 조회 | 가능 |
| 코칭 메시지 관리 | 가능 |
| 라이더 노출 ON/OFF 변경 | 가능 |
| 내부 메모 작성 | 가능 |
| 페이스 체크 설정 변경 | 가능 |
| 운영 로그 조회 | 가능 |
| 정산 관리 앱 링크 이동 | 가능 |
| 라이더 본인 수기 입력 대리 작성 | 기본 불가, 별도 운영 정책 필요 |

관리자 계정 운영 원칙:

- 관리자 수는 최소화한다.
- 관리자 권한 부여/회수 이력을 남긴다.
- 관리자 화면 접근은 개인 계정 기준으로 추적한다.
- 공용 관리자 계정은 운영 공개 후 사용하지 않는다.

### rider

라이더는 본인 `rider_id`에 매핑된 데이터만 조회한다.

| 기능 | 권한 |
| --- | --- |
| 본인 내현황 조회 | 가능 |
| 본인 내오더 조회 | 가능 |
| 본인 내지도 조회 | 가능 |
| 본인 내코칭 조회 | 가능 |
| 오늘의 페이스 체크 수기 입력 | 가능 |
| 본인 페이스 체크 입력 수정 | 가능 |
| 정산앱 링크 이동 | 가능 |
| 관리자 화면 접근 | 불가 |
| 타 라이더 선택 UI | 불가 |
| 타 라이더 오더/집계/코칭 조회 | 불가 |
| 업로드/검수/코칭 관리 | 불가 |
| 운영 로그 조회 | 불가 |

라이더 화면 원칙:

- 최신 반영 주차 기준 데이터를 보여준다.
- 실시간 관제, 현재 위치, 진행중 오더처럼 보이는 표현을 쓰지 않는다.
- 라이더가 볼 수 있는 데이터는 본인 `rider_id` 기준으로 제한한다.

### pending

승인대기 계정은 승인대기 안내만 조회한다.

| 기능 | 권한 |
| --- | --- |
| 승인대기 안내 조회 | 가능 |
| 관리자 화면 접근 | 불가 |
| 라이더 데이터 화면 접근 | 불가 |
| 본인 오더/코칭 조회 | 불가 |
| 오늘의 페이스 체크 입력 | 불가 |
| 운영 로그 조회 | 불가 |
| 정산앱 링크 접근 | 운영 정책으로 결정, 기본은 불가 권장 |

pending 운영 원칙:

- pending 상태에서는 운행 데이터와 코칭 데이터가 노출되지 않는다.
- active 전환 시 반드시 `rider_id` 매핑을 확인한다.
- 잘못 매핑된 `rider_id`는 타 라이더 데이터 노출로 이어질 수 있으므로 승인 단계에서 검수한다.

## `rider_id` 매핑 정책

`rider_id`는 코칭센터에서 라이더 계정과 주차별 운행 데이터를 연결하는 핵심 키다.

운영 기준:

- 라이더 개인 계정 1개는 하나의 고정 `rider_id`를 가진다.
- 엑셀 업로드 데이터의 라이더명은 운영 매핑 기준에 따라 `rider_id`로 연결한다.
- 라이더 이름이 바뀌어도 `rider_id`는 유지한다.
- 퇴사/이탈/비활성 계정의 과거 `rider_id`를 다른 라이더에게 재사용하지 않는다.
- 라이더 로그인 후 조회되는 내현황/내오더/내지도/내코칭은 본인 `rider_id` 데이터만 사용한다.

운영 전 결정할 항목:

- `rider_id`를 기존 정산앱과 동일하게 맞출지 여부
- 동명이인 처리 방식
- 라이더명 오타 또는 변경 시 관리자 수정 절차
- 계정 비활성화 후 과거 데이터 보관 기간

## 정산앱과의 계정 정책

초기 베타와 1차 운영에서는 정산앱과 코칭센터를 각각 로그인하는 방식을 권장한다.

정책:

- 정산앱과 코칭센터의 자동 로그인/SSO는 이번 베타 범위에서 제외한다.
- 코칭센터의 정산앱 링크는 단순 이동 링크다.
- 정산앱 인증, 권한, 정산 상세, 지급 상태, 확정 처리는 정산앱에서 처리한다.
- 코칭센터는 정산 계산, 정산 엑셀 파싱, 지급 상태 판단을 하지 않는다.
- 향후 DB 도입 후 공통 `rider_id` 기준의 느슨한 연동을 검토할 수 있다.
- 계정 통합이나 SSO는 개인정보 처리 범위, 권한 책임, 장애 대응 기준을 별도로 확정한 뒤 진행한다.

## 향후 Supabase/Auth 도입 시 고려사항

Supabase/Auth를 도입할 경우 아래 항목은 별도 설계와 보안 검토가 필요하다. 실제 구현 전에는 Supabase 공식 문서와 changelog를 다시 확인한다.

검토 항목:

- `users` 또는 `profiles` 테이블
- Supabase Auth 사용자 ID와 앱 사용자 프로필 매핑
- `rider_id` 고정 매핑
- `role` 기반 접근 제어
- `account_status` 기반 승인대기 분리
- Row Level Security 정책
- 관리자/라이더/승인대기 권한 분리
- 비밀번호 재설정 흐름
- 감사 로그
- 테스트 계정 제거 또는 seed 데이터 분리

Supabase RLS 방향 초안:

| 테이블 | admin | rider | pending |
| --- | --- | --- | --- |
| `profiles` | 전체 조회/승인 상태 관리 | 본인 프로필 조회 | 본인 승인대기 상태 조회 |
| `upload_weeks` | 전체 조회/관리 | 본인 데이터가 있는 최신 반영 주차 메타 조회 | 접근 불가 |
| `normalized_orders` | 전체 조회/관리 | 본인 `rider_id` 데이터만 조회 | 접근 불가 |
| `rider_weekly_summaries` | 전체 조회/관리 | 본인 `rider_id` 집계만 조회 | 접근 불가 |
| `coaching_messages` | 전체 조회/수정 | 본인 `rider_id`이고 `visible_to_rider = true`인 메시지만 조회 | 접근 불가 |
| `pace_check_entries` | 운영 정책에 따라 제한 조회 | 본인 입력 생성/수정/조회 | 접근 불가 |
| `pace_check_settings` | 조회/수정 | 읽기 전용 조회 | 접근 불가 |
| `operation_logs` | 전체 조회 | 접근 불가 | 접근 불가 |

Supabase/Auth 주의사항:

- 노출 schema의 테이블은 RLS를 켠다.
- 브라우저 클라이언트에 `service_role` 또는 secret key를 노출하지 않는다.
- 권한 판단에는 사용자가 수정할 수 있는 `user_metadata`를 사용하지 않는다.
- `raw_app_meta_data` 또는 DB 프로필 테이블 기반 권한 모델을 검토한다.
- `auth.uid()`가 없는 비로그인 요청은 데이터 접근이 실패하도록 명확히 정책을 작성한다.
- view, security definer function, Data API 노출 설정은 별도 보안 검토가 필요하다.

참고:

- Supabase Row Level Security 문서: https://supabase.com/docs/guides/database/postgres/row-level-security
- Supabase User Management 문서: https://supabase.com/docs/guides/auth/managing-user-data

## 운영 전 최종 원칙

1. 라이더는 본인 데이터만 본다.
2. 관리자는 전체 데이터를 볼 수 있지만 변경 이력은 남긴다.
3. 승인대기 계정은 데이터 화면에 접근하지 못한다.
4. 테스트 계정은 운영 공개 전에 숨기거나 제거한다.
5. 정산앱은 별도 시스템이며 코칭센터와 권한이 통합된 것이 아니다.
6. 원천 엑셀 파일과 불필요한 개인정보는 저장하지 않는다.
