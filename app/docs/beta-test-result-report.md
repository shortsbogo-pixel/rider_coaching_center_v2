# 라이더 코칭센터 Ver.2 1차 베타 테스트 결과 보고서

## 1. 베타 테스트 개요

| 항목 | 내용 |
| --- | --- |
| 테스트 목적 | 라이더 코칭센터 Ver.2의 관리자/라이더 핵심 흐름, 엑셀 업로드 기반 코칭 구조, 정산앱 링크 연결, 모바일 사용성을 1차 베타 범위에서 검증한다. |
| 테스트 대상 | 관리자, 라이더, 승인대기 계정, 내부 운영자, 휴대폰 사용자 |
| 테스트 방식 | Vercel 배포 URL에 접속해 테스트 계정으로 화면 흐름을 확인하고 피드백을 수집한다. |
| 배포 URL | https://rider-coaching-center-v2.vercel.app |
| 정산앱 연결 URL | https://ridersettle-fmyozrbw.manus.space/ |
| 테스트 계정 | `admin / admin1234`, `rider1 / rider1234`, `pending / pending1234` |
| 테스트 범위 | 로그인, 권한 분기, 관리자 업로드/검수/코칭/운영 로그, 라이더 내현황/내오더/내지도/내코칭/MY, 오늘의 페이스 체크, 정산앱 링크, 모바일 360px 확인 |

이번 베타는 정식 운영 공개가 아니라 소규모 검수용이다. 현재 구조는 DB 저장 없이 브라우저 세션/in-memory 상태를 중심으로 동작하며, 실제 개인정보나 원천 엑셀 파일을 코드/배포물에 포함하지 않는다.

## 2. 테스트 대상

| 대상 | 확인 목적 |
| --- | --- |
| 관리자 | 업로드, 검수, 반영, 대시보드, 코칭 메시지, 운영 로그, 페이스 체크 설정, 정산앱 링크 확인 |
| 라이더 | 본인 데이터 조회, 오늘의 페이스 체크, 내오더/내지도/내코칭/MY, 정산앱 링크 확인 |
| 승인대기 계정 | 승인대기 화면만 표시되고 관리자/라이더 데이터 화면에 접근하지 못하는지 확인 |
| 내부 운영자 | 베타 진행 순서, 오류 캡처, 피드백 분류, 공개 판단 확인 |
| 휴대폰 사용자 | 모바일 360px 폭 기준 가독성, 버튼 터치, 하단 메뉴, 카드 겹침 여부 확인 |

## 3. 테스트 계정

| 역할 | ID | 비밀번호 | 확인 범위 |
| --- | --- | --- | --- |
| 관리자 | `admin` | `admin1234` | 관리자 전체 메뉴 |
| 라이더 | `rider1` | `rider1234` | 본인 라이더 화면 |
| 승인대기 | `pending` | `pending1234` | 승인대기 화면 |

테스트 계정은 베타 검수용이다. 운영 전에는 테스트 계정 안내를 숨기거나 계정을 폐기해야 한다. 실제 운영 계정과 테스트 계정 비밀번호를 공유하지 않는다.

## 6. 정산앱 링크 테스트 결과

| 확인 항목 | 결과 | 메모 |
| --- | --- | --- |
| 코칭센터 → 정산앱 | 확인 필요 | 라이더 MY, 관리자 더보기에서 확인 |
| 정산앱 → 코칭센터 | 확인 필요 | 정산앱 관리자 링크 설정 기준 |
| 새 탭 열림 | 확인 필요 | 기존 화면을 잃지 않아야 함 |
| 링크 노출/비노출 설정 | 확인 필요 | 정산앱 관리자 모드 기준 |
| 데이터 연동 아님 | 확인 필요 | 단순 링크 이동임을 문구로 확인 |

정산 계산, 지급, 확정, 상세 원장은 정산앱 기준이다. 운행 분석, 코칭, 오늘의 페이스 체크는 코칭센터 기준이다. 두 서비스는 데이터 연동, SSO, 권한 통합을 하지 않는다.

| 정산앱 데이터 직접 연동 여부 | 초기에는 링크 유지 권장 | 링크 유지 권장 |
| 개인정보 최소화 기준 | 전화번호/상세주소/계좌정보 등 제외 | 미정 |
| 운영 담당자 | 업로드/검수/피드백 담당자 지정 필요 | 미정 |
| 오류 발생 시 대응 방식 | 긴급/높음/보통/낮음 기준으로 분류 | 미정 |

## 13. DB/Supabase 도입 판단

### 현재 DB 없이 가능한 범위

- 관리자/라이더/pending 권한 분기 화면 확인
- 브라우저 파일 선택 방식의 엑셀 파싱
- 업로드 미리보기와 확정 반영 흐름 확인
- 관리자 대시보드/검수/코칭/운영 로그 UI 확인
- 라이더 본인 화면과 오늘의 페이스 체크 UX 확인
- 정산앱 외부 링크 이동 확인

### DB가 필요한 이유

- 새로고침 후 데이터 유지
- 여러 사용자/기기에서 같은 최신 주차 데이터 공유
- 라이더 개인 계정과 `rider_id` 고정 매핑
- 코칭 메시지와 관리자 수정 이력 보관
- 오늘의 페이스 체크 기록 저장
- 운영 로그 장기 보관
- RLS 기반 라이더 본인 데이터 접근 제어

### Supabase 도입 장점

- PostgreSQL 기반으로 주차/오더/집계/코칭 데이터를 구조화하기 좋다.
- Supabase Auth와 RLS를 통해 관리자/라이더/pending 권한 분리를 설계할 수 있다.
- 운영 로그, 계정 상태, 주차별 업로드 이력 보관에 적합하다.
- 향후 백업, 감사, 권한 관리 체계를 만들기 쉽다.

### Supabase 도입 리스크

- RLS 정책이 잘못되면 타 라이더 데이터 노출 위험이 있다.
- Auth, 계정 발급, 비밀번호 재설정, 관리자 권한 관리가 필요하다.
- 비용, 백업, 운영 담당자, 장애 대응 기준이 필요하다.
- 원천 엑셀 파일을 저장하지 않는 정책을 계속 지켜야 한다.
- 정산앱과 직접 연동할 경우 책임 영역과 개인정보 범위가 커진다.

### 도입 전 필요한 준비

- 사용자/계정 정책 확정
- `rider_id` 매핑 정책 확정
- 저장/비저장 데이터 범위 확정
- RLS 정책 초안과 테스트 케이스 작성
- 운영 로그 보관 기간 결정
- 백업/복구 정책 결정
- 테스트 계정과 실제 운영 계정 분리

### 도입 추천 시점

| 시점 | 추천 |
| --- | --- |
| 단기 | DB 없이 1차/2차 베타 운영 가능 |
| 중기 | Supabase/Auth/DB 설계 후 도입 검토 |
| 정식 운영 전 | 사용자 계정과 주차 데이터 저장 구조 필요 |

판단 질문:

- 베타 사용자가 늘어나는가?
- 매주 업로드 데이터를 계속 보관해야 하는가?
- 라이더 개인 계정이 필요한가?
- 코칭 메시지 저장이 필요한가?
- 오늘의 페이스 체크 기록 저장이 필요한가?
- 관리자 운영 로그를 영구 보관해야 하는가?

## 14. 최종 판단

| 판단 | 조건 | 이번 상태에서 의미 |
| --- | --- | --- |
| A. 공개 가능 | 로그인 정상, 관리자/라이더 권한 정상, 업로드/검수/반영 정상, 정산앱 링크 정상, 모바일 주요 화면 정상, 긴급 이슈 없음 | 소규모 베타 참여자에게 공유 가능 |
| B. 일부 보완 후 공개 | 핵심 기능은 정상이나 일부 문구/UX/안내 보완 필요, DB는 없어도 베타 목적에는 문제 없음 | 보완 항목을 기록하고 제한된 베타로 공개 가능 |
| C. 공개 보류 | 로그인 불가, 권한 오류, 타 라이더 데이터 노출, 업로드/반영 불가, 모바일 주요 화면 사용 불가, 정산앱 링크 실패 | 원인 수정 전까지 URL 공유 보류 |

현재 상태 기준 권장 판단:

**소규모 베타 공개 가능**

단, DB/Supabase가 아직 도입되지 않았으므로 정식 운영 공개가 아니라 1차/2차 베타 운영 범위로 제한한다. 새로고침 후 재업로드 필요, 테스트 계정 사용, 원천 엑셀 미저장, 정산앱은 단순 링크 이동이라는 제한을 베타 참여자에게 명확히 안내해야 한다.

## 15. 다음 작업 제안

1. 긴급/높음 이슈 처리
   - 로그인, 권한, 데이터 노출, 업로드/반영, 정산앱 링크 문제를 우선 확인한다.

2. 2차 개발 후보 확정
   - DB 저장, 개인 계정, 코칭 메시지 저장, 페이스 체크 기록 저장, 알림/문자, 지도 고도화 등을 후보로 정리한다.

3. 로그인/개인 계정 전환 여부 결정
   - 테스트 계정을 계속 쓸지, 개인 계정 체계로 전환할지 결정한다.

4. Supabase 도입 여부 결정
   - 사용자 수, 데이터 보관 필요성, 권한/RLS 필요성을 기준으로 판단한다.

5. 운영 버전 로드맵 작성
   - 1차 베타, 2차 베타, 정식 운영 전환 단계별 기능/운영 목표를 분리한다.
